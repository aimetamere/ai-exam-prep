Model + dependencies + config → reproducible artifact. Docker + ONNX + model card. Anything not rebuildable from a git tag is a liability in prod.

MLOps · Retraining & Versioning
Automated Retraining — Triggers, Gates & Safe Promotion
Retraining on a schedule is a lazy default. Trigger-based retraining is the right answer.
Retraining trigger strategies
❌ Schedule-based (naive)
Retrain every Monday at 2am. Simple. Wasteful when data hasn't changed. Dangerous when drift happens mid-week and you wait 6 days to respond.
✅ Trigger-based (correct)
Retrain when: (1) PSI exceeds threshold, (2) model accuracy drops below SLA, (3) new labelled data volume exceeds N, or (4) a business event occurs — regulatory change, product launch.
💡 Tip: combine both. Use trigger-based as primary, schedule-based as a backstop. If no trigger fires in 30 days, force a retrain anyway. Drift detectors can have blind spots.

Model versioning with MLflow
Stages: None → Staging → Production → Archived. Never manually overwrite a Production model. Always promote through staging. Every version is queryable by name + version string.
🔮 Champion/challenger pattern: Run the new challenger model on 5–10% of real traffic alongside the champion. Collect live performance data for 24–48h. Only promote if challenger beats champion on all KPIs and business metrics. This catches what offline eval misses.
Rollback strategy
Always keep N-1 model in Staging, ready to promote back. Rollback must be a one-command operation, not a full pipeline re-run.

LLMOps · Foundations
Why LLMOps is Fundamentally Different from MLOps
LLMs break every assumption that classical MLOps was designed around
What classical MLOps assumes — and LLMs violate
AssumptionReality with LLMs
Deterministic outputsSame prompt → different output every time. Testing is probabilistic.
Hard error failuresLLMs fail silently — hallucinations look like correct answers.
Versioning = swap a filePrompt changes are code. Model weights + system prompt = version.
Latency in millisecondsLLM inference: 500ms–30s. Token streaming changes UX architecture.
Cost = compute timeCost = tokens in × price + tokens out × price. Easy 100× cost spikes.
Eval = accuracy scoreCorrectness + safety + groundedness + consistency + helpfulness.
The 5 production failure modes
1. Hallucination
Confident, fluent, completely wrong. Hard to detect without grounding or eval harnesses. Catastrophic in regulated industries.
2. Prompt Injection
User input hijacks system prompt: "Ignore previous instructions." Treat all user input as untrusted. Sanitise before it reaches the model.
3. Silent Model Drift
Provider updates the base model. Your prompts subtly break. Users complain. You find out two weeks later.
4. Context Overflow
Long conversation exceeds context window. Model silently drops the system prompt — the most important part of the context.
5. Cost Explosion
One uncapped prompt × 1M users = Monday morning bill shock. Always set max_tokens. Always.

LLMOps · Evaluation
LLM Evaluation — You Cannot A/B Test Your Way to Safety
Evals must be systematic, automated, and run on every single prompt change
The five evaluation dimensions
Correctness
Is the answer factually right? Build your ground-truth dataset from production Q&A logs with expert annotation. Expensive once, invaluable always.
Groundedness (RAG)
Is every claim supported by retrieved context? Ragas scores faithfulness, context precision, and recall. A grounded answer that contradicts its source is still a hallucination.
Safety & Robustness
Prompt injection resistance, jailbreak resistance, content policy compliance. Test with a real red-team dataset — not toy examples.
Consistency
Same answer to 5 paraphrases of the same question? High variance = reliability risk. Test paraphrase sets for every critical query type.
Latency & Cost
p95 latency within SLA? Cost per query within budget? These are eval dimensions too — a correct but 10-second answer fails a real-time SLA.
Eval tools — honest comparison

HeliconeProd monitoringGateway-level tracing, no SDK changes
PromptfooRed-teamingBuilt-in adversarial + jailbreak library
💡 LLM-as-Judge: Use GPT-4o or Claude to evaluate outputs from smaller models. Scales without human labellers. But calibrate the judge against human labels first — uncalibrated judges introduce systematic biases that correlate with style, not correctness.
🔮 Eval-driven development: Write your eval suite before writing your prompt. Define "good" in measurable terms first. Then write the prompt to pass the evals. Same discipline as TDD — dramatically better output quality.

Vector databases
Pinecone — managed, scalable Weaviate — open source, hybrid Qdrant — fast, on-prem friendly PGVector — already in Postgres Chroma — local dev only
💡 Hybrid search is non-negotiable in production. Dense vector search misses exact keyword matches (product codes, IDs). BM25 misses semantic similarity. Use both, fuse with Reciprocal Rank Fusion. Improves recall by 15–25% with minimal latency overhead.
🔮 Reranker as quality multiplier. After vector search returns top-20, run a cross-encoder reranker (Cohere Rerank, BGE-Reranker) to reorder by true relevance. ~50ms overhead, 25–35% precision improvement. LLM answer quality is directly proportional to top-3 chunk quality.
⚠️ Embedding model mismatch. You must use the same embedding model at index time and query time. Switching models = full re-index. Choose carefully before building the index. all-mpnet-base-v2 is a safe default for English.
💡 PGVector before specialised vector DB. If under ~1M vectors, PGVector gives you vector search inside Postgres. No new infra, ACID compliance, no new ops burden. Migrate only when you actually outgrow it.

LLMOps · Control Layer
Guardrails, Rate Limits & Cost Controls
Three layers every production LLM deployment must have before the first user ever sees it
🛡 Guardrails
  • PII detection and redaction before model sees input
  • Prompt injection detection — regex + semantic classifiers
  • Topic blocking — refuse out-of-scope queries gracefully
  • Output filtering — harmful content, sensitive data leakage
  • Groundedness check — flag responses contradicting context
  • Toxicity scoring on every output before returning to user
NeMo GuardrailsGuardrails AIBedrock GuardrailsLlama Guard
⏱ Rate Limits
  • Per-user token budgets per day (prevent abuse)
  • Per-tenant request quotas (multi-tenant fairness)
  • Model-level TPM/RPM caps (respect provider limits)
  • Graceful degradation: queue then reject, never hang
  • Burst allowance with token bucket algorithm
  • Priority queues: paid users over free tier
🔮 Pattern: API Gateway → Redis token counter → LLM. Never expose LLM endpoint directly to users.
💰 Cost Controls
  • max_tokens on every request — no exceptions, ever
  • Daily/monthly spend hard cap per environment and tenant
  • Model router: cheap model first, flagship only when needed
  • Prompt caching for static system prompts and documents
  • Semantic cache: return cached response for near-duplicate queries
  • Kill switch: auto-disable if spend >2× 7-day rolling average
💡 Semantic caching with GPTCache or Momento: 30–40% of queries in customer support are near-duplicates. This is free money.

LLMOps · Monitoring
Production LLM Monitoring — What to Measure and How
If you instrument nothing else, instrument these six things first
The six essential metrics
Latency p95/p99
Time-to-first-token (TTFT) and total generation time. With streaming, users tolerate slow completion if TTFT is fast.
Hallucination rate
Automated groundedness scoring on a sample of prod responses. Track weekly. Any upward trend = investigate immediately.
Cost per query
Input + output tokens × price. Track by tenant, feature, model. Cost spikes at 3am are a real thing.
User signal
Thumbs-down rate, session abandonment, follow-up "try again" rate. Best proxy for quality without ground truth.
Guardrail hit rate
Spike in PII detection = data leak risk. Spike in topic blocking = prompt injection wave in progress.
Context utilisation
Consistently >80% = risk of context overflow silent failures. Time to add context compression middleware.
Monitoring architecture
Trace every LLM call: prompt, response, model, tokens in/out, latency, guardrail results, user ID, session ID

Async evaluation: run groundedness + safety eval on 5–10% of prod calls in a background queue — never block the user response

Apply anomaly detection to LLM metrics: the same time-series anomaly detection concepts apply here — cost spikes, latency spikes, and hallucination rate increases are all detectable anomalies
💡 OpenLLMetry: OpenTelemetry-native instrumentation for LLM calls. One decorator → full trace propagation from your API layer through every LLM call, tool call, and retrieval step. Works with LangSmith, Datadog, Honeycomb, any OTLP backend.
🔮 Shadow evals in prod: Run your eval suite nightly against a random sample of real production conversations — not the happy path. The messy, adversarial, real stuff your test suite never anticipated. This is where actual failure modes live.
⚠️ PII in traces: LLM traces contain raw user inputs. PII ends up in your logs unless you actively prevent it. Run PII scrubbing before trace data leaves the application tier. GDPR auditors will find it if you don't.

Agentic Ops · Foundations
What is an Agentic System — And Why It Breaks Everything
An agent doesn't just answer — it plans, decides, uses tools, and acts across multiple steps
The anatomy of an agent
  • Planner — LLM that decides what steps to take next
  • Memory — short-term (context window) + long-term (vector store)
  • Tool use — web search, code execution, database queries, API calls
  • Executor — runs the tool, returns result to the planner
  • Loop — repeats until task complete or stop condition triggered
Business examples today
  • 🏦 Procurement agent — queries suppliers, generates POs, approves payments
  • 📋 Research agent — searches web, reads papers, writes structured summaries
  • 🛡 Security agent — monitors alerts, runs playbooks, patches systems
  • 💬 Customer ops agent — accesses CRM, processes refunds, escalates complaints
  • 💻 Coding agent — reads codebase, writes tests, opens pull requests
Why agents break classical monitoring
Non-deterministic
Same input → different action sequence every time. You cannot replay and reproduce failures reliably. Classical debugging doesn't work.
Multi-step with compounding errors
Failure at step 7 of 12 may have already taken irreversible actions at steps 2–6. By the time you detect it, the damage is done.
Tool-using — real blast radius
The agent can write code, call APIs, send emails, modify databases, approve payments. Blast radius is real, immediate, and sometimes irreversible.
The fundamental shift: Classical monitoring asks "did the function return the right value?" Agentic Ops asks "did the agent take the right sequence of actions, at the right confidence level, with appropriate oversight at each step?"

Agentic Ops · Risk Architecture
The 5 Failure Modes of Autonomous Agents
Each requires a specific operational guardrail — not just monitoring after the fact
1. Hallucinated Tool Calls
Agent calls a function with made-up parameters. The tool executes with bad data. Records corrupted, transactions submitted, emails sent.
Guardrail
JSON schema validation on every tool input before execution. Dry-run mode for destructive tools.
2. Infinite Loops
Agent retries a failing tool indefinitely. Loop runs until token budget or billing limit is hit. Common when tool returns ambiguous errors.
Guardrail
Max step count. Max retry per tool. Circuit breaker on repeated identical actions.
3. Scope Creep
Agent "helpfully" takes additional actions beyond the task definition. Small expansions compound into large unintended consequences.
Guardrail
Strict task definition. Allowlist of permitted tools per task type. Human approval for any tool call outside the list.
4. Context Overflow
Long task sequences exceed context window. Agent "forgets" original instructions and constraints mid-task. Behaviour becomes unpredictable.
Guardrail
Persistent memory store. Re-inject system prompt every N steps. Context compression middleware.
5. Conflicting Instructions
Tool output contradicts original instructions. Agent oscillates between two valid-but-opposed actions, making no progress and accumulating cost.
Guardrail
Explicit conflict resolution rules. Escalate to human when confidence drops below threshold.
Design principle: Design for failure, not success. Every agentic system must have a documented answer to "What happens when the agent does something wrong — and how do we roll it back?" before it goes to production.
🔮 Critic + Actor pattern: Run a second LLM as a "critic" that evaluates every proposed action before the executor runs it. The critic checks: is this action within scope? Is the confidence high enough? Does this contradict earlier instructions? The overhead is ~30% but dramatically reduces failure modes 1, 3, and 5.

Agentic Ops · Control Architecture
Trust Boundaries, Kill Switches & Audit Trails
The three non-negotiable components of any production agent deployment
🔐 Trust Boundary
  • Explicit allowlist of tools the agent can call — no open-ended access
  • Scope permissions with IAM/RBAC — agent identity ≠ admin identity, ever
  • Human-in-the-loop checkpoints for high-risk actions before execution
  • Human-on-the-loop monitoring for medium-risk autonomous decisions
  • Blast radius: maximum damage if agent acts incorrectly must be bounded and reversible
🛑 Kill Switches
  • Emergency stop that halts all agent activity immediately — one command
  • Circuit breakers that auto-pause on anomalous action sequences
  • Automatic suspension if cost/token budget exceeded by >2×
  • Business-hour restrictions for financial and operational agents
  • Automatic pause if same tool fails more than N times in M minutes
📋 Audit Trails
  • Immutable log of every action: what was called, with what params, at what time
  • Full reasoning chain: why the agent made each decision (LLM reasoning trace)
  • Input/output hashes for integrity verification — detect tampering
  • Required for: GDPR Article 22, EU AI Act, financial audit, SOC2 compliance
  • Minimum retention: 90 days for most financial use cases; 7 years for some

Agentic Ops · Advanced Patterns
Multi-Agent Systems — Patterns, Coordination & New Failure Modes
When one agent isn't enough — and the operational complexity multiplies non-linearly
Multi-agent architecture patterns
Orchestrator + Subagents
One orchestrator delegates tasks to specialist agents. Orchestrator holds the goal; subagents hold the tools. Clean separation of concerns — the most common and most reliable pattern.
Peer-to-peer agents
Multiple agents of equal authority negotiate and check each other's work. Useful for adversarial red-team/blue-team setups or parallel exploration of solution spaces.
Critic + Actor pattern
Actor proposes actions. Critic evaluates them before the Executor runs them. Dramatically reduces hallucinated tool calls and scope creep. ~30% overhead cost, worth it for high-risk actions.
💡 Tip: don't build multi-agent until a single agent is reliable and fully observable. Complexity compounds. Start simple — add agents only when the single-agent bottleneck is provable.
New failure modes unique to multi-agent
  • Agent collusion: Two agents agree on an incorrect plan with no external check. The orchestrator trusts both — no one catches the error.
  • Contradictory actions: Agent A creates a record; Agent B deletes it simultaneously. Race conditions in distributed agent systems are very real.
  • Trust propagation: A compromised or hallucinating subagent poisons the orchestrator's context with bad information that propagates to subsequent steps.
  • Attribution failure: An incident occurs — which agent made the bad decision? Shared tool credentials make forensics impossible.
  • Emergent behaviour: The interaction between agents produces outcomes that neither agent would produce alone and that no one anticipated in design.
🔮 Ops rule: Every agent in a multi-agent system must have its own identity, its own audit trail, and its own blast-radius boundary. Shared identity = shared failure mode. No exceptions — not even for subagents that "only do read operations."

Agentic Ops · Tips & Pitfalls
What the Documentation Doesn't Tell You About Agents
Hard-won patterns from real agentic deployments — most of them painful to learn
Tips that move the needle
  • Tool schemas are your most important prompt. A well-documented tool schema with clear param descriptions, types, and examples reduces hallucinated tool calls by 60–80%. Treat tool documentation like production API documentation.
  • Start with read-only tools. Build confidence in the agent's planning quality before giving it write access. Agents that can only read are observable, debuggable, and recoverable. Add write tools one at a time with clear risk assessment for each.
  • Use Temporal for anything that touches money or data. Temporal's durable execution model means crashed agents resume from their last checkpoint — not from the beginning. For financial agents, this prevents double-execution of payments on recovery.
  • Add a "confidence" field to every tool output. Tell the agent how much to trust each tool result. Search engines can return irrelevant results. APIs can return stale data. A confidence score lets the agent decide when to verify before acting.
  • Log the full reasoning chain, not just actions. When debugging agent failures, the reasoning trace — the "thinking" before each tool call — is more valuable than the tool call itself. Always capture it, even if it costs extra tokens.
Pitfalls that sink agent projects
  • Giving agents too many tools. Every additional tool increases the probability of choosing the wrong one. Start with the minimum set needed for the task. A focused agent with 5 tools outperforms a capable agent with 50 tools.
  • No dry-run mode for destructive operations. The agent should always be able to simulate an action and show what it would do before doing it. No dry-run = no safe way to test in staging with production-like tasks.
  • Trusting agent self-reporting on confidence. LLMs are notoriously overconfident. Don't ask the agent "are you sure?" — it will say yes. Instead, require independent verification for high-stakes actions: a second tool call that validates the first result.
  • No rollback strategy for write operations. Every write operation an agent performs needs a defined undo procedure. "Delete this record" needs "restore this record." "Send this email" needs an archive + retraction workflow. Design rollback first.
  • Testing only the happy path. Agents fail in adversarial, ambiguous, and unexpected conditions — not the clean test cases you wrote. Red-team your own agent with edge cases, conflicting instructions, and deliberately bad tool outputs before go-live.

Agentic Ops · Maturity Model
Agentic Ops Maturity — From Assisted to Fully Autonomous
Most enterprises are at Level 1. Level 4 is the frontier. Governance gaps are the primary bottleneck.
Level 1 — Assisted
Human initiates every task. Agent provides suggestions, drafts, or analysis. Human reviews and approves every output before any action is taken.
Examples
Copilot tools, drafting assistants, code suggestion, research summarisation
Where most orgs are
~60% of deployments
Level 2 — Supervised Automation
Agent completes multi-step tasks autonomously. Human reviews and approves at defined checkpoints. Agent cannot act without explicit approval on gated steps.
Examples
Ticket triage + draft response (human sends), invoice matching (human approves payment)
Where leaders are
~25% of deployments
Level 3 — Conditional Autonomy
Agent acts autonomously within clearly defined boundaries. Human-on-the-loop for anomalies. Automatic escalation when confidence drops or action type changes.
Examples
Auto-approve low-value POs, auto-resolve known incident types, auto-respond to standard customer queries
~12% of deployments
Level 4 — Full Autonomy
Agent operates independently across complex, open-ended tasks. Self-corrects, self-monitors, requests human help only when genuinely stuck. Kill switches + full audit required.
Examples
End-to-end software engineering agents, autonomous trading within defined risk parameters
~3% — frontier territory
⚠️ The governance gap: Most organisations skip from Level 1 directly to Level 3 because the technology is ready. The governance, legal, and compliance frameworks are not. EU AI Act Article 14 requires human oversight for high-risk AI systems. Define your risk classification before choosing your autonomy level.
💡 Earn autonomy incrementally. Deploy at Level 1. Measure. Show the data. Negotiate Level 2 with legal and compliance. Measure again. Every jump in autonomy level needs to be earned with data — not assumed because the demo worked. This is how you build lasting organisational trust in agentic systems.
